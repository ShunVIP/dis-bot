# Контекст для разработчика/агента

1. Пародии в ViPik строго Markov-only (`мем`, `разум`). Не возвращать GPT, Transformers, Persona или remote model bridge.
2. ML добавляется в другие подсистемы: модерация, активность, экономика, игры, качество данных и ранжирование итогов.
3. Новые модели сначала работают в shadow/recommendation режиме. Санкции, списания и блокировки требуют проверенной разметки и отдельной метрики качества.
4. Тяжёлое обучение выполняется на ПК; VPS предназначен для SQLite, лёгких артефактов и inference. Допустимо обучать на VPS только действительно лёгкий пайплайн, если production-данные нельзя безопасно экспортировать.
5. Хранилища и ML-логи живут в `core/`, Discord cogs отвечают за UI/orchestration.
6. Пользовательские данные должны иметь один канонический store для Discord, админки и web/app.
7. Перед deploy: unit tests, compile, `git diff --check`; после deploy: systemd status и журналы обоих сервисов.
8. Production: `/opt/dis-bot`, units `vipik-discord-bot.service` и `vipik-web-app.service`.
9. Не коммитить `KGTD.env`, production SQLite, токены, временные bundles и локальные настройки.
10. Следить за лимитом глобальных Discord slash-команд; настройки по возможности объединять в группы или переносить в приватную админку.
