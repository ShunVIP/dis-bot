# AGENT_CONTEXT

Этот файл предназначен для другого ИИ-агента или нового человека, который впервые открывает проект.

## Что это за проект

`ViPik Discord Bot` — Discord-бот с играми, экономикой, статистикой, напоминаниями, Steam-интеграцией и системой пародии на стиль речи.

Это не просто локальный бот. У проекта есть рабочая прод-схема:

- код редактируется на ПК;
- прод-бот постоянно работает на VPS;
- `messages.db` собирается на VPS и является главным источником сообщений;
- безопасный `Markov` обучается на VPS автоматически;
- тяжёлая `GPT`-модель обучается только локально на ПК;
- VPS использует тяжёлую GPT-модель через bridge с ПК, если bridge включён.

## Главные правила

1. Не коммить секреты, токены, `KGTD.env`, базы, `models/`, SSH-ключи.
2. Не запускать GPT-обучение на VPS.
3. Не ломать ежедневный цикл `сбор сообщений -> Markov` на VPS.
4. Не считать локальную `messages.db` источником правды, пока она не синхронизирована с VPS.
5. Не переносить тяжёлую GPT-модель на VPS.
6. Любые изменения в `systemd`, `bridge`, `Tailscale`, env-флагах и деплое делать аккуратно.

## Где что живёт

### ПК

- разработка кода;
- `git`;
- локальное GPT-обучение;
- локальные модели в `models/`;
- запуск bridge;
- центр управления.

### VPS

- прод-бот;
- `systemd`-сервис `vipik-discord-bot`;
- `KGTD.env`;
- боевые базы;
- ежедневный сбор сообщений;
- ежедневный безопасный `Markov`;
- веб-панель.

## Источник правды

- главная база сообщений: `messages.db` на VPS;
- локальная копия на ПК нужна только для обучения и анализа;
- если локальная база не синхронизирована, нельзя считать её актуальной.

## Как работает GPT

### Обучение

- GPT обучается только на ПК;
- обучение берёт сообщения из локальной копии `messages.db`;
- обученная модель остаётся на ПК.

### Использование на VPS

- VPS не хранит тяжёлую GPT-модель;
- когда bridge включён, VPS обращается к модели на ПК по `Tailscale`;
- если ПК выключен или bridge выключен, тяжёлая GPT недоступна, но бот продолжает работать.

## Что можно и нельзя на VPS

### Можно

- обычную работу бота;
- безопасный ежедневный `Markov`;
- ежедневный сбор сообщений;
- веб-панель;
- использование bridge.

### Нельзя

- GPT-обучение;
- тяжёлую полную профилактику;
- хранение тяжёлой GPT-модели.

## Главные файлы

- `main_file.py` — запуск бота
- `config.py` — env
- `core/runtime_policy.py` — серверные ограничения и флаги
- `core/admin_panel.py` — веб-панель
- `scripts/центр_управления.ps1` — локальный центр управления
- `scripts/train_models_menu.ps1` — локальный сценарий обучения
- `scripts/train_local.py` — локальный training runner
- `scripts/enable_remote_models.ps1` — включение bridge для VPS
- `scripts/disable_remote_models.ps1` — выключение bridge
- `scripts/sync_messages_from_vps.ps1` — скачать свежую `messages.db`
- `scripts/sync_training_to_vps.ps1` — отправить лёгкие артефакты на VPS

## Главный интерфейс обслуживания

Использовать:

- `Центр управления ботом.bat`

Актуальные пункты:

- `3` — скачать свежую `messages.db`
- `4` — обучить модели локально
- `5` — отправить лёгкие артефакты на VPS
- `9` — включить тяжёлую модель с ПК для VPS
- `10` — выключить тяжёлую модель
- `11` — `git status`
- `12` — commit + push
- `14` — установка нового VPS
- `15` — статус бота на VPS
- `18` — `git pull origin main`

## Веб-панель

Назначение:

- смотреть статус;
- смотреть защитную политику;
- включать/выключать использование удалённой тяжёлой модели.

Не предназначена для:

- локального GPT-обучения;
- локального `git`;
- синхронизации базы на ПК;
- локального управления bridge-процессом.

Открывается через `Tailscale`, а не публичный интернет.

## Если нужно работать безопасно

Перед изменениями:

1. прочитать `README.md`
2. прочитать этот файл
3. не трогать реальные секреты
4. не ломать `.gitignore`
5. не ломать runtime policy

После изменений:

1. проверить синтаксис/явные ошибки
2. если менялся прод-код — проверить сервис на VPS
3. кратко описать, что изменено

## Если проект передали архивом

Новый агент должен:

1. прочитать `README.md`
2. прочитать `AGENT_CONTEXT.md`
3. понять, что прод живёт на VPS, а GPT — на ПК
4. не пытаться "упростить" это, таща GPT на VPS
5. использовать центр управления как главный локальный интерфейс
