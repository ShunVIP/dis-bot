# 🧠 ARCHITECTURE — dis-bot v2.0 Stable

Документ описывает **архитектуру стабильного билда dis-bot v2.0**.

⚠️ Всё, что здесь описано, считается **фундаментом проекта**.
Изменение существующих механизмов **запрещено** — допускаются только надстройки.

---

## 🎯 Архитектурные цели

Архитектура dis-bot v2.0 спроектирована так, чтобы:

* быть **модульной** (каждый домен — отдельный Cog);
* иметь **минимальные зависимости между модулями**;
* использовать **локальное хранилище (SQLite)** без внешних сервисов;
* позволять добавлять новые функции **без изменения существующих**;
* быть устойчивой к рестартам бота.

---

## 🚀 Точка входа

### `main_file.py`

Ответственность:

* запуск бота;
* инициализация логирования;
* настройка intents;
* автозагрузка всех cogs;
* запуск планировщиков.

❌ Не содержит бизнес-логики.

---

## 🧩 Cogs — модульная логика

Каждый файл в `fun_slesh/`:

* реализует **один домен**;
* сам отвечает за свои таблицы БД (`CREATE TABLE IF NOT EXISTS`);
* регистрирует **slash-команды**.

### Основные cogs

| Cog                          | Ответственность             |
| ---------------------------- | --------------------------- |
| `daily.py`                   | Дэйлик, серии, баланс, топы |
| `games.py`                   | Игры PvE / PvP              |
| `rep_and_mood.py`            | Репутация и настроение      |
| `birthday.py`                | Дни рождения                |
| `tools.py`                   | Напоминания, счётчики       |
| `mem_roles.py`               | Временные роли              |
| `ai_tools.py`                | Wikipedia, PubMed           |
| `achievements_engine.py`     | Ачивки                      |
| `message_and_voice_stats.py` | Статистика                  |
| `test_hello.py`              | Фан, опросы, инфо           |

Принцип:

> **Один cog — одна зона ответственности**

## 🔔 Event Bus (ключевой элемент)

### `utils/events_bus.py`

Event Bus — это **мягкая связка между модулями**.

### Зачем он нужен

* Игры не знают об ачивках
* Дэйлик не знает о наградах
* Репутация не знает о достижениях

Вместо этого:

```python
await emit("daily_claimed", user_id=..., streak=...)
```

И подписчик:

```python
subscribe("daily_claimed", on_daily_claimed)
```

### Гарантии

* события выполняются **последовательно**;
* ошибки обработчиков **не валят бота**;
* нет циклических зависимостей.

⚠️ В v2.0 **запрещено**:

* прямое обращение одного cog к логике другого;
* выдача ачивок напрямую, минуя события.

## 💰 Domain Services

### Экономика (`core/economy.py`)

* единая точка работы с монетами;
* журнал транзакций (ledger);
* защита от silent-изменений баланса.

Все начисления проходят через:

```python
add_coins(user_id, delta, reason, meta)
```

---

## 🏅 Achievements Engine

### `fun_slesh/achievements_engine.py`

* подписывается на события;
* содержит **только правила** выдачи;
* не зависит от UI или команд.

Архитектурное правило:

> Ачивки **реактивны**, а не инициирующие.

---

## 🗄 Базы данных

### Принцип

* SQLite — основной источник истины;
* каждая БД — отдельный домен;
* схемы создаются лениво.

### Используемые БД

| БД             | Назначение                           |
| -------------- | ------------------------------------ |
| `social.db`    | экономика, ачивки, игры, статистика  |
| `reminders.db` | напоминания, таймеры, временные роли |
| `birthdays.db` | дни рождения                         |

❌ Запрещено:

* cross-db JOIN;
* shared foreign keys между БД.

---

## ⏰ Планировщики

Используется **APScheduler**:

* напоминания;
* автоснятие ролей;
* поздравления с ДР.

Особенности:

* таймзоны: **MSK / UTC**;
* задачи восстанавливаются из БД после рестарта.
